# Hook-Layer Defect Audit — claude-infrastructure

**Scope:** ~40 `hooks/*.sh` + live `~/.claude/settings.json` wiring. Read-only. **No stdin-eater exists** (the prior-incident class is fully absent — every hook reads stdin once into a var; the two `python -` heredocs feed script via `<<'PY'` not hook JSON; mailbox-drain deliberately `cat >/dev/null` to avoid SIGPIPE). **No hook can hang a turn indefinitely** (untimed ones hold no lock/network/prompt-wait; bounded by the 60s default). The real damage is **leaks + latency + inert mechanisms**, several empirically confirmed live.

## DEFECT ROWS

**[S1] Unbounded watchdog-daemon leak — no single-instance guard, no process reaper**
· File: `hooks/lead-crash-watchdog.sh:95-101,113-123,268-269`
· Mechanism: Every SessionStart spawns a detached `( while :; sleep 30; kill -0 $LEAD_PID ) & disown` daemon. Nothing checks whether a daemon already runs for this session/pid before spawning. Exit depends on the pid-file vanishing or `kill -0` failing — both defeated by (a) PID reuse across macOS sleep/wake (`kill -0` succeeds forever against a recycled pid → immortal daemon) and (b) `set -e` (line 16) aborting the daemon mid-loop before cleanup → orphaned pid-file. `team-orphan-reaper.sh:61` removes pid *files* only for team-associated sids and never `kill`s the bash processes; solo-session daemons are unreaped.
· Failure scenario: Machine sleeps with a session open → wake → daemon polls a reused pid → never exits. Accretes one immortal process per affected start; contributes to the documented single-process-OOM crash history (memory: session-crash-per-session-bloat).
· Evidence: **21 live daemons right now**, oldest etime `07-06:04:43` (7 days), several multi-day; **105 orphaned `.pid` files** (sampled pids DEAD); **1918 `cp-*.count` files**, 2167 total entries in `~/.claude/watchdog/`. `while :; ... sleep 30` (line 113/122); `) </dev/null >/dev/null 2>&1 &` + `disown` (268-269); no `pgrep`/guard before spawn (grep-confirmed absent).
· Fix sketch: Before spawn, `pgrep -f "lead-crash-watchdog.*$SESSION_ID"` guard; store spawn-pid and validate identity (not just `kill -0`) — e.g. compare `ps -o lstart`/comm; add a launchd reaper that kills daemons whose pid-file is gone or whose lead comm no longer matches `claude`.
· Class: leak · Status: **NEW** (git log: abdafc4 added death-classification, no lifecycle guard) · Confidence: **empirical**

**[S2] `waiting-recycle.sh` — only un-timeouted hook on the hot PostToolUse Bash slot; desk path spawns 10-30 subprocs per Bash call**
· File: `hooks/waiting-recycle.sh:493-563`; wiring `settings.json:533` (no `timeout`)
· Mechanism: On the monitoring desk (operator's primary session), every Bash tool call runs the full eval: multiple `git` calls (rev-parse/status/git-dir, 493-497), a `$CFG/teams/session-*` glob (532), a per-file loop over `wait-contracts/*.json` doing **4 `jq` calls each** (541-546), a per-file loop over `cc-roles/*` (557-562), plus `stat` mailbox checks. Cost scales with accumulated coordination-file count. Slot-sibling `log-bash.sh` has `timeout:5`; this one relies on the 60s default — a single wedged subprocess stalls the Bash tool up to 60s.
· Failure scenario: Stale `wait-contracts/` accumulate (same non-reaping pattern as the pid files) → per-Bash-call `jq` fan-out grows unbounded → visible desk latency; a hung `git`/`stat` on a slow FS stalls the turn 60s with no per-hook cap.

· Class: stall · Status: **NEW** · Confidence: **empirical** (cost) / **theoretical** (60s wedge)

· File: `hooks/{completion-assert,anti-deference-nudge,boundary-handoff,waiting-recycle}.sh` (each `>> "$IDL"` per Stop/Bash); `hooks/teammate-checkpoint.sh:46,171-178`

**[S2] `teammate-auto-shutdown.sh` detached-close child can orphan a hung iterm2 probe + leave the worktree unreaped**
· File: `hooks/teammate-auto-shutdown.sh:453-477`, `_pane_from_tty:186-213`
· Mechanism: After emitting `{"continue":false}`, the hook spawns a detached `( sleep 3; close pane; worktree remove ) &` with **no hook-timeout backstop** (it's already returned). Inside, `_pane_from_tty` runs `iterm2.run_until_complete(main)`; the 3s guard is `asyncio.wait_for(_find(),3)` which wraps only the enumeration — **not the websocket connection setup**. If the iTerm2 API socket stalls, the connection blocks before `main` runs → the orphaned python hangs indefinitely, and because it hangs before line 470, `git worktree remove` never executes → leaked worktree.
· Failure scenario: iTerm2 busy/unresponsive at reap time → orphaned python process + a worktree that accumulates until manual GC.
· Evidence: `) >/dev/null 2>&1 &` (477, no disown/no timeout); `LATE_PANE=$(_pane_from_tty ...)` (462); `sid = await asyncio.wait_for(_find(), timeout=3)` wraps only `_find` (203), while `iterm2.run_until_complete(main)` (209) is unguarded.
· Fix sketch: Wrap the detached child in `timeout 15 …`; move worktree-remove BEFORE the pane close (or make it unconditional on the child's exit).
· Class: leak · Status: **NEW** · Confidence: **theoretical**

**[S2] Unescaped shell→JSON interpolation in `cat <<EOF` decision emitters (user-derived values)**
· File: `hooks/validate-bash.sh:46-57,152` (lead); also `agent-teams-enforce.sh:54-59` (`$MODEL`), `backup-before-write.sh:95-102` (`$BASENAME`/`$FILE`)
· Mechanism: `warn()`/`deny()` build permission-decision JSON via a double-quoted heredoc, interpolating `$1`. Line 152 passes user-derived `$target` (extracted from the Bash command, `:147`) into the reason. A `rm -rf` target containing `"`, `\`, or a newline emits **malformed JSON** → Claude Code can't parse the `permissionDecision` → the intended `ask` is lost and the `rm` proceeds under default flow. All hardcoded-string reasons are safe; only user/tool-derived interpolations break.
· Failure scenario: `rm -rf 'a"b'` (or a path with a backslash) on a non-safe target → warn JSON malformed → confirmation skipped. Same class silently drops the OVERWRITE GUARD (backup-before-write) or the model-allowlist deny (agent-teams-enforce) on odd inputs.
· Evidence: `"permissionDecisionReason": "$1"` inside `cat <<EOF` (39/52) with `warn "rm -rf on non-build-artifact target: '$target'. …"` (152); contrast the hardened hooks' `jq -nc --arg`.
· Fix sketch: Replace the heredocs with `jq -nc --arg r "$1" '{hookSpecificOutput:{hookEventName:"PreToolUse",permissionDecision:"ask",permissionDecisionReason:$r}}'`.
· Class: closure · Status: **NEW** · Confidence: **theoretical** (defense-in-depth: settings deny-list + rm-safe-allowlist still catch catastrophic rm)

**[S3] `CLAUDE_CONTINUE_MAX` cap is bypassable by design**
· File: `hooks/session-continue.sh:66,138-146,175`
· Mechanism: The "hard loop cap" (default 8) is reset to 0 by any fresh `set` (`rm -f "${f}.count"`, line 66), and the block reason it feeds back (175) explicitly instructs the model to re-arm via `set` each 🔧 turn. An obedient model re-arms every turn → `.count` never reaches MAX → the runaway guard never bites. It only catches a model that STOPS re-arming.
· Failure scenario: A model stuck genuinely believing work remains keeps re-arming → unbounded continuation loop; the cap provides no backstop against the obedient-but-wrong case.
· Evidence: `rm -f "${f}.count"` on set (66); "a fresh set zeroes .count — this is how a long grind stays under the ${MAX}-cap" (175).
· Fix sketch: Track a *monotonic* per-cwd continuation counter that `set` does NOT reset (separate from the re-arm counter); cap on the monotonic one.
· Class: closure · Status: **NEW** (self-documented as intentional) · Confidence: **theoretical**

**[S3] Empty-sid arming defeats SID-BIND → stale sentinel blocks a successor up to 8 stops**
