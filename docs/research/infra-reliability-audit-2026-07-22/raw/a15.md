## (2) DEFECT ROWS

**[S2] session-index mkdir-fallback lock has no stale reap → a SIGKILL/OOM permanently wedges all session indexing**
- **File:** `hooks/lib/session-index-helpers.sh:88-99` (`session_index_trylock`, the wired path) — corroborated by `:72-79`.
- **Mechanism:** `flock` is absent on this box (verified), so `trylock` always takes `mkdir "$LOCKFILE.d"`. The `.d` path has **no TTL and no pid-liveness check** anywhere, and no cron/reaper removes it (`grep session-index.lock` across hooks/bin/scripts/launchd = empty). `unlock` only `rmdir`s on a clean EXIT trap, which **does not fire on SIGKILL**.
- **Failure scenario:** `com.claude.session-search-sweep` (launchd timer) acquires the `.d` lock, then OOM-kills mid-parse (the helper *itself* guards against OOM by skipping >50 MB transcripts at `:473` — i.e. this is the repo's most OOM-exposed process, and OOM is a live event here per commit `ef7f7a4`). The `.d` dir is orphaned forever. Every subsequent sweep **and** every `session-index-end.sh` (`:31`) calls `trylock` → `mkdir` fails → returns 1 → "skip if locked". **Session indexing silently stops until a human `rmdir`s the dir.**
- **Evidence:** `while ! mkdir "$SESSION_INDEX_LOCKFILE.d" 2>/dev/null; do … if [ "$_attempts" -ge 300 ]; then … return 0` (`:72-77`); `if mkdir "$SESSION_INDEX_LOCKFILE.d" 2>/dev/null; then … return 0; fi; return 1` (`:94-99`).
- **Fix sketch:** give the `.d` path the same stale recovery as L1/L2 — check dir mtime age (or a written pid via `kill -0`) and reclaim; or a launchd `find "$LOCKFILE.d" -maxdepth 0 -mmin +10 -exec rmdir {} \;`.
- **Class:** stall · **Status:** NEW · **Confidence:** empirical (flock-absence + no-reaper verified; blast radius reasoned).

**[S2] `cc-backlog compact` is an unlocked read-rewrite-`mv` → a concurrent append is lost**
- **File:** `bin/cc-backlog:320-335`.
- **Mechanism:** cc-backlog is otherwise safe by append-only discipline, but `compact` reads the whole file into `$tmp`, then `mv "$tmp" "$BACKLOG"`. There is **no lock anywhere in cc-backlog**. Any record appended between the read loop and the `mv` is discarded by the overwrite.
- **Failure scenario:** Actor A = human runs `cc-backlog compact`; reads N lines into `$tmp`. Actor B = `cc-dispatch` tick (or `cc-reaper` reap on the `com.claude.dispatcher` timer) appends a `claim` record (line N+1). A's `mv` overwrites the file → the claim vanishes → the item folds back to `status:"open"` → the next dispatch tick **claims and spawns a second worker** on in-flight work (the exact 2026-07-20 double-dispatch class the reopen-guards at `:216-246` were added to stop, reintroduced through the compaction door).
- **Evidence:** `tmp=$(mktemp) … done < "$BACKLOG"` then `mv "$tmp" "$BACKLOG"` (`:321-335`) with no surrounding lock; contrast the append writers `>> "$BACKLOG"` (`:187,254`).
- **Mitigant (lowers probability, not severity):** `grep` finds **no daemon/cron caller** of `compact` — it is manual-only, so the window requires a human running it during a dispatch/reap tick.
- **Fix sketch:** wrap `compact` in a land-lock-style pid lock (or `flock` where present), or make it append a compaction-marker rather than rewrite.
- **Class:** race (lost update) · **Status:** NEW · **Confidence:** empirical (code path) / theoretical (interleave).

**[S3→S2 if ever wired] `session_index_lock` 300s-timeout path returns success without the lock and without arming its trap**
- **File:** `hooks/lib/session-index-helpers.sh:74-76`.
- **Mechanism:** on the blocking path, `_SESSION_INDEX_LOCK_FD=9` is set at `:68` *before* the flock attempt. When flock is absent and the mkdir loop times out at 300 attempts, it `return 0` (success) — but this returns **before** `trap 'session_index_unlock' EXIT` at `:82`, and leaves `_FD="9"` (not "mkdir"). The caller believes it holds the lock (rc 0) while another holder's `.d` still exists, and a later `session_index_unlock` runs the wrong branch (`flock -u 9` no-op instead of `rmdir`).
- **Failure scenario:** two backfill/tagger processes both time out, both `return 0`, both write the DB concurrently. **Currently latent** — `grep` shows only `trylock` callers (`session-index-sweep.sh:26`, `session-index-end.sh:31`); nothing calls the blocking `session_index_lock`. Backstopped even if wired: the DB itself runs `PRAGMA journal_mode=WAL` + `busy_timeout` (`:161-163`, `:44`) so the outcome is wasted work / `SQLITE_BUSY`, not corruption.
- **Evidence:** `if [ "$_attempts" -ge 300 ]; then session_index_log "…proceeding anyway"; return 0; fi` (`:74-76`).
- **Fix sketch:** on timeout `return 1` (let caller skip), or set `_FD="mkdir"` + arm the trap before proceeding.
- **Class:** race · **Status:** NEW (latent) · **Confidence:** empirical (unreachable today), theoretical (impact).

**[S3] mtime-TTL mkdir locks steal + double-release (no owner verification)**
- **File:** `hooks/lib/mailbox-pending.sh:72,78`; `bin/cc-idl:70,77`.
- **Mechanism:** both reap purely on the lock dir's mtime age (10s / 60s) and both `unlock` with an **unconditional `rm -rf`** (`_mbx_unlock`, `release_lock`) — no owner token is written or verified. A holder that overruns the TTL has its lock stolen by a second actor; the original holder's unlock then deletes the *thief's* lock, admitting a third.
- **Failure scenario:** Actor A holds the mailbox lock; is SIGSTOP'd / load-starved past 10s. Actor B sees `age ≥ stale`, `rm -rf`s, `mkdir`s, enters. A resumes, `_mbx_unlock` `rm -rf`s B's lock; Actor C now `mkdir`s and also enters → two concurrent cursor writers → a **duplicate** mail delivery. Bounded to dup (never loss): the design at `:118-136` accepts dup over hang, and the guard keys on the `.acked` cursor (`:9-11`) so drops stay loud. cc-idl's variant is moot — it has **zero callers** (whole-repo grep empty), so its chain is effectively never sealed/contended.
- **Evidence:** `[ "$age" -ge "$stale" ] && { rm -rf "$ld"; continue; }` (`mailbox:72`); `_mbx_unlock() { rm -rf "$(_mbx_dir)/.${1:-}.lock" …` (`mailbox:78`). Same shape at `cc-idl:70,77`.
- **Also folds in:** `mailbox_take`/`mailbox_promote_acked` proceed **lock-free** after the 2 s give-up (`:127,141`) — two lock-free takers both read `prev`, both print, both write `seen=EOF` → same bounded dup. Explicitly intended ("dup-risk, never a hang").
- **Fix sketch:** write `$$`/uuid into the lock dir; on unlock, `rmdir` only if the token matches. (Or accept — the channel is dup-tolerant by contract.)
- **Class:** race · **Status:** NEW · **Confidence:** theoretical (needs a >TTL stall of a millisecond-scale critical section).

**[S3] `live-session-registry.sh` writes the liveness row with truncate-then-write, not tmp+mv**
- **File:** `hooks/live-session-registry.sh:55`.
- **Mechanism:** register does `printf '%s\t%s\t%s\n' … > "$REG_DIR/$base"`. `>` opens `O_TRUNC` (file → 0 bytes) as a separate syscall before the single `write()`. A reader in that gap sees an empty file. This is the **only** non-atomic shared write in a codebase that otherwise religiously uses tmp+mv — including its sibling `session-register.sh:81` (`mv -f "$tmp" "$reg_dir/$pane.json"`) and `cc-decide`.
- **Failure scenario:** worktree-gc reaper reads `cut -f2 "$REG_DIR/$base"` during a re-register's truncate→write gap → empty pid → `kill -0 ""` fails → the **live** worktree reads as dead → reaped. This re-introduces (at ~one-printf width) the exact single-bad-read flakiness the file's own header (`:5-10`) says it exists to eliminate. Self-limiting: `cc-reconcile` re-creates a missing live row.
- **Evidence:** `printf … > "$REG_DIR/$base" 2>/dev/null` (`:55`) vs `mv -f "$tmp" "$reg_dir/$pane.json"` (`session-register.sh:81`).
- **Fix sketch:** `printf … > "$REG_DIR/.$base.$$" && mv -f "$REG_DIR/.$base.$$" "$REG_DIR/$base"`.
- **Class:** race · **Status:** NEW · **Confidence:** theoretical.

**[S3] `session-end.sh` version-GC lock has no stale reap and no trap (shares a dir with a stronger lock)**
- **File:** `hooks/session-end.sh:19,53`.
- **Mechanism:** `mkdir "$lock_dir" 2>/dev/null || exit 0` then `echo $$ > "$lock_dir/pid"`, released only by a bare `rm -rf` at `:53` (no trap, so SIGKILL orphans it) and **never** pid-liveness-checked. It shares `~/.claude-versions/.cleanup_lock` (confirmed `VERSIONS_DIR=$HOME/.claude-versions` in both) with `claude-latest`, which *does* reclaim by `kill -0` (`:99-105`).
- **Failure scenario:** session-end GC is SIGKILLed holding `.cleanup_lock`; every future SessionEnd GC does `mkdir → fail → exit 0` (silent skip). Blast radius is only this *secondary* GC (stale `~/.claude-versions` dirs = disk), and it self-heals the moment any `claude-latest` launch runs its pid-liveness reclaim on the same dir.
- **Evidence:** `mkdir "$lock_dir" 2>/dev/null || exit 0` (`:19`), `rm -rf "$lock_dir"` (`:53`), no `trap`.
- **Fix sketch:** copy `claude-latest`'s pid-liveness reclaim block; add `trap 'rm -rf "$lock_dir"' EXIT`.
- **Class:** leak/stall · **Status:** NEW · **Confidence:** empirical (shared dir + asymmetric policy verified).

**[S3] `claude-accounts --fresh` uses a blocking flock with no wall-clock cap**
- **File:** `bin/claude-accounts:806` and `:837`.
- **Mechanism:** for `allow_degrade=False` (the `--fresh` path) `_acquire_lock` does a bare `fcntl.flock(lock, LOCK_EX)` with no deadline; the non-fresh path correctly uses `LOCK_NB` + a 5s deadline + stale-cache grace (`:828-838`). The holder "can own this for MINUTES" (the code's own comment) — 4 accounts × keychain + a 90s heal subprocess + retries.
- **Failure scenario:** a `--fresh` holder wedges *alive* (hung heal / network stall). A second `--fresh` caller — `handoff-fire.sh` runs `claude-accounts --fresh --json` before firing a wave — blocks indefinitely, stalling the wave. Bounded by kernel release on holder **death** (fcntl) and (assumed) heal-subprocess timeout, so only an alive-but-hung holder stalls; no deadlock cycle (the only other flock, per-acct heal L13, is `LOCK_NB`).
- **Evidence:** `if not allow_degrade: fcntl.flock(lock, fcntl.LOCK_EX); return True` (`:805-806`); fallback `fcntl.flock(lock, fcntl.LOCK_EX)` (`:837`).
- **Fix sketch:** give the fresh path a wall-clock deadline too, then a loud "serving stale cache — fresh sweep wedged" fallback.
- **Class:** stall · **Status:** NEW · **Confidence:** theoretical.

## (3) CHECKED-CLEAN

- **`scripts/land-lock.sh`** — the model: keyed on `--git-common-dir` (shared across worktrees, `:42-45`) so all worktrees collide on one mutex; **pid-liveness reap** (live holder *never* stolen, dead reaped immediately, `:88-107`); `trap release EXIT` + reap backstop robust to SIGKILL. **This closes the documented 2026-07-11 sibling-`/ship` rebase-drop incident.** Tested (`tests/land-lock.bats`).
- **`bin/claude-latest`** install + cleanup locks — pid-file + `kill -0` reclaim (`:100-105`, `:236-241`); SIGKILL-robust without a trap.
- **`bin/cc-decide`** — file-per-decision + tmp+`mv` atomic rename (`:109-120`,`:143-150`,`:197-201`); idempotent open never overwrites evidence (`:96`). No shared-file RMW.
- **`hooks/session-register.sh`** — pane-uuid rows via `mv -f "$tmp"` (`:81`), atomic. **`bin/cc-reconcile`** — additive/stale-heal only, never deletes a live row, fail-closed prune when `it2` unreadable.
- **`mailbox_migrate` two-box lock** — acquired in fixed lexicographic order (`:226-228`), deadlock-free by construction; append-before-advance is the no-loss guarantee (`:235-253`). Cursor semantics tested (`mailbox-drain.bats:73` append-during-drain no-loss-no-dup).
- **`claude-accounts` non-fresh cache lock (L12 degrade path) + heal lock (L13)** — `LOCK_NB` + deadline + stale-cache-grace; real fcntl → auto-release on death. Coordinates relogin vs heal so no account double-login.
- **`reset-hard-shadow-allow.sh`** — in-process atomic 3-conjunct state check (shape + clean-tree + **sanctioned linked worktree, never the shared checkout**), fail-closed, shadow-by-default. Sound answer to the "24/7 autonomy strands on the reset-hard `ask` prompt" problem without a keystroke-approval TOCTOU.
- **`git-worktree-guard.sh`** — blocks branch-delete-with-worktree and worktree-remove-with-live-cwd (pgrep+lsof), fail-open on parse error. Backstops the janitor for manual git.
- **Append-only stores** (`idl.jsonl`, `cc-backlog` add/transition `:187,254`, `cc-route` route.jsonl `:70`, `cc-respawn` `:45`, per-session `.fired` sets) — bare O_APPEND; **empirically atomic** (live IDL max line 309 B < 512 PIPE_BUF; 370 K lines, no chronic malformed-line reports from `cc-backlog`'s fail-loud validator `:88-104`). *Caveat (not a defect today):* atomicity is size-dependent and unenforced — a future >512 B line (long `--needs` / `what_plain` / discovery title) from two concurrent daemons could tear.
- **Lock ordering** — the only two-lock sites are `mailbox_migrate` (ordered) and `claude-accounts` cache→heal (ordered, heal is NB); nothing acquires them in reverse → **no deadlock cycle**. No Stop hook takes a blocking lock (flock absent; Stop hooks append bare `>>`; the 2 s-bounded mailbox lock degrades lock-free; `cc-idl`'s 10 s-wait lock is wired to nothing).

## Uncertainties / named blockers
- **cc-idl blast radius rests on "zero callers."** Whole-repo grep (excl. self/tests) is empty and no launchd job seals — but a P3 **supervisor** referenced in `cc-idl`'s header may live outside this checkout. If a periodic sealer exists elsewhere, D4's cc-idl variant rises from moot to a real (still dup-bounded) contention.
- **D7 hinges on the heal subprocess being timeout-bounded** — I did not read the heal body's timeout; if heal can hang unbounded while alive, D7's stall is unbounded for concurrent `--fresh` callers.
- **O_APPEND atomicity on APFS** is empirical (no tearing in 370 K live lines), not POSIX-guaranteed for regular files — PIPE_BUF technically governs pipes. The 512 B headroom is the practical margin, and it is unenforced.
