# Cross-Component Architecture-Invariants Review — claude-infrastructure

Derived invariants first (before code), then verified at seams. Every verified claim carries file:line; derivations-only are marked theoretical.

## (1) INVARIANT SET — what a self-managing multi-session fleet must hold

| # | Invariant | The load-bearing requirement |
|---|---|---|
| **I1** | Identity continuity across recycle | A logical identity (role, mail address, claim) must survive a sessionId/pid swap. Addressing keys on a **stable handle** (pane UUID / role file), never the ephemeral sessionId or pid. |
| **I2** | At-most-once, successor-verified pane close | A pane closes only after a live successor is verified **and** its WIP is landed/checkpointed; identity re-pinned at act-time; no two actors close one pane. |
| **I3** | Exactly-one singleton per role | desk / dispatcher / poller / supervisor = exactly one live instance. Recycle & spawn must transiently create neither 0 (gap) nor 2 (split-brain). |
| **I4** | At-most-once / idempotent message effect | A mail/nudge/dispatch takes effect once; redelivery is idempotent; the dedup cursor is keyed on the **event**, not a mutable surface. |
| **I5** | Bounded persistent state | Transcripts, ledgers, registries, marker dirs are reaped/rotated. **The recycle-TRIGGER metric must correlate with the OOM-CAUSE metric.** |
| **I6** | Crash containment | One death must not wedge a lock, tear a shared file, or strand a claim/marker. Writes atomic (tmp+rename); locks self-breaking; timestamps a shared clock. |
| **I7** | Automation acts on a fresh world-view | Observe→act gaps are revalidated at act-time (identity pin) or locked; change-detection keys on **fixed** reference points, not moving baselines. |
| **I8** | Liveness ≠ registration (dual-truth reconciliation) | A row can outlive its process; a process can run unregistered. The ledger's liveness key must point at the **actual worker**, not a proxy. |
| **I9** | Deliver only to a ready receiver | Typed delivery must not land in a modal / mid-turn / reused pane; use a context-insensitive channel or verify readiness; an unanswerable modal on an unattended session is surfaced. |
| **I10** | Spawn atomicity / no partial actors | A spawn yields a fully-engaged, registered actor or none. "Fired" must mean "engaged"; partial waves are reconciled. |

## Seams the brief flagged that are actually WELL-DEFENDED (deprioritize follow-up)

- **reaper × handoff (I2/I7):** fully guarded. cc-classify emits `{pid,lstart}` (cc-classify:386–396) → cc-reaper pins `--expect-pid/--expect-lstart` (cc-reaper:432–437) → cc-teardown **REFUSES** on mismatch (cc-teardown:199–206), RED-proven (selftest #13, cc-teardown:468–479). A mid-transition pane is not killed.
- **recycle × mailbox / identity continuity (I1):** well-handled. Mail is **pane-UUID-addressed** (stable across in-place `--recycle`); succession repoints roles (`refresh_roles_for`) + writes the `.forward` pointer **before** the self-close watcher is armed (handoff-fire:1114→1116→1143), with SessionStart adoption as a backstop (mailbox-drain:61–78). No "closing pane still addressed" window.
- **pane-typed delivery × modal (I9):** well-handled for messaging. cc-notify v2 is **non-keystroke** — it appends to a file inbox drained at safe boundaries (cc-notify:18–31), so a modal just *delays* delivery, never corrupts input. Permission hangs get an unspoofable harness beacon → supervisor page (cc-permission-beacon.sh:60–73; lead-supervisor:417–450).

## (2) VIOLATION ROWS (verified in code)

**V1 · [S2] I8/I10 · seam: cc-dispatch × cc-backlog-reap × cc-reaper — the claim ledger is worker-liveness-blind**
- **Evidence:** cc-dispatch claims `--by "$SID"` where `SID=<host>-$$` = the **cron dispatcher's** pid (cc-dispatch:58,261), which exits seconds after spawning. `cc-backlog reap` gates on `claimAgeS > 5400s AND !claimer_live` (cc-backlog:447–448); `claimAgeS = now − last-claim-ts` (cc-backlog:420, a monotonic clock, not worker activity); `claimer_live` does `kill -0` on that dead dispatcher pid (cc-backlog:357–359). The worker never re-claims under its own identity (dispatch brief cc-dispatch:275–281).
- **Mechanism:** Because `by` is the ephemeral dispatcher, `claimer_live` is ~always false for a dispatched item, so reap decides **purely on claim-age**. Nothing in the ledger observes the actual worker. The 90-min gate is a heuristic tuned to p90=55min (cc-backlog:376–377) — it cannot see a *live* long task or a *dead* short one.
- **Failure scenario:** a deep-research/large-build worker still running at minute 91 → reap reopens → next tick re-claims → **second worker onto in-progress work** → both race to land → rebase exit 5 → thrash → item BLOCKED (Rule B, cc-backlog:437–443) after burning two sessions. Symmetric failure: a worker that dies at minute 5 strands its item claimed for 85 min.
- **Fix:** link the claim to the worker, not the dispatcher — have the fired session re-claim `--by <own paneUUID/sid>` on engagement (handoff-fire already verifies engagement at :2101), or record the worker paneUUID in the claim and make `claimer_live` resolve it via cc-sessions (the MEMORY "check worktree liveness" rule), or worker-heartbeat the claim.
- **Confidence: empirical** — double-dispatch incident family documented (2026-07-20 a60d62a215f1→6488617); the >90-min path is a direct code reading.

**V2 · [S2] I5 · seam: waiting-recycle/boundary-handoff × OS process — recycle-trigger vs OOM-cause metric decorrelation**
- **Evidence:** recycle fires on **context `used_pct`** only (waiting-recycle:51,71; boundary-handoff:124). No transcript-byte trigger exists in either recycle hook (grep for `stat -f %z`/`wc -c` on transcripts hits only session-index + tests). The crash fix ef7f7a4 landed *detection* (lead-crash-watchdog + cc-crash-report), not a size-triggered recycle.
- **Mechanism:** `used_pct` measures the **live context window**; the OOM cause is **transcript bytes** (on-disk JSONL / process memory). They decorrelate — many turns with large tool outputs grow bytes while old turns fall *out* of context, so `used_pct` stays moderate. The recycle threshold is never crossed → the giant session never recycles → single-process OOM.
- **Failure scenario:** a long desk/worker greps/reads large files across hundreds of turns, hits ~60MB+ transcript, and is OOM-killed while `used_pct` sits ~40% (below every tier) — "crash to shell", no recycle hook ever fires.
- **Fix:** add a transcript-byte/turn-count trigger to the recycle predicate (`stat -f %z <transcript> ≥ ceiling` → force size-recycle, independent of `used_pct`); the watchdog already computes the size for reporting — wire it into waiting-recycle's fire predicate.
- **Confidence: empirical** — documented OOM crash + MEMORY session-crash-per-session-bloat; code confirms recycle is `used_pct`-only.

**V3 · [S2] I9 · seam: lead-supervisor × statusline-telemetry × live panes — the supervisor's world-view can be empty while sessions are live**
- **Evidence:** `sweep()` iterates `$TEL_DIR/*.json` only (lead-supervisor:452–467); DEAD/STALL?/PAST-THRESHOLD all key on telemetry rows (:365,385,394). `TEL_DIR=/tmp/cc-telemetry` (:44) — cleared on reboot; the statusline stops emitting for a non-rendering/backgrounded/long-turn session (supervisor's own note :271–284). Unlike **cc-reaper**, which has an INDEPENDENT `live_pane_count` self-check vs enumerated (cc-reaper:216–223,308–335), the supervisor has **no** telemetry-rows↔live-panes reconciliation.
- **Mechanism:** a live session with no telemetry row (post-reboot before first render, broken/backgrounded statusline, `/tmp` cleared) is invisible to every telemetry-driven supervisor path — it can hang or sit past-threshold and never be paged. The reaper sees it (registry/it2); the supervisor does not — **divergent visibility of the same fleet**.
- **Failure scenario:** after reboot, boot-resume relaunches sessions; one hangs before its statusline emits; the supervisor sweeps an empty `/tmp/cc-telemetry`, heartbeats `swept=0`, pages nothing.
- **Fix:** give the supervisor the reaper's blind-spot twin — compare telemetry-row count to an independent live-claude-pane count (reuse `live_pane_count`/cc-sessions), page on positive delta. (Permission-pending is already telemetry-independent via the beacon — extend that principle to stall/threshold.)
- **Confidence: theoretical** — code-verified structural gap; the statusline-goes-stale premise is the codebase's own (supervisor:271–284), but the empty-telemetry-with-live-panes event isn't directly observed here.

**V4 · [S3→S2 latent] I4/I6 · seam: cc-notify × mailbox-drain × cc-await-ping — exactly-once degrades to at-least-once**
- **Evidence:** the single-lock snapshot+advance gives exactly-once (mailbox-pending.sh:18–21), **but** the mkdir-lock self-breaks after ~2s and "degrades to lock-free (risking a benign DUP, never a hang)"; `mailbox_take` proceeds on lock-giveup (`_mbx_lock "$u" || true`, :127). mailbox-drain (SessionStart) and an armed cc-await-ping watcher poll the **same** inbox.
- **Mechanism:** under lock contention >2s (slow disk, or two drainers + a migration on one box), a drainer proceeds lock-free and can re-surface an already-advanced line → a **dup**. Benign for *advisory* context (the model re-reads), but the invariant is "exactly-once **effect**".
- **Failure scenario:** today — a peer sees one inbox line twice after a lock-timeout (harmless). **Latent:** if the mail channel is ever used imperatively (e.g., a nudge that triggers spawn/claim/land), the dup double-acts.
- **Fix:** keep the channel advisory-only by contract (lint that bodies are never imperative), or add an application-level message-id the consumer dedupes on, making at-least-once transport safe for any effect.
- **Confidence: empirical** (the degradation is explicitly coded + commented); imperative-dup harm is theoretical (no imperative use today).

**V5 · [S3] I2/I7 · seam: cc-teardown × handoff-fire (in-place recycle) — the identity pin is strong but OPT-IN**
- **Evidence:** the pin protects the *reaper* path only (passed conditionally, cc-reaper:435–436). cc-teardown's `resolve()` re-reads the pane's **current** row (cc-teardown:148–159); its ACT-leg lstart check treats a recycle as **success** (`proc_gone=1`, cc-teardown:261–266), detecting the swap only *after* the kill. A caller that resolves-then-acts **without** `--expect-*` (a desk running bare `cc-teardown <pane>`, or classify emitting an empty lstart) has only this post-hoc check.
- **Mechanism:** if the pane recycles between a manual classify and the un-pinned call, `resolve()` picks up the **successor's** pid; the kill hits the successor; the lstart check then reports a "verified teardown" of the freshly-booted successor.
- **Failure scenario:** a desk tears down a pane it saw idle 30s ago; in that window the pane `/handoff --recycle`'d in place; the un-pinned teardown kills the successor and reports success.
- **Fix:** make the pin **intrinsic** — cc-teardown captures pid+lstart at `resolve()` and requires they still match at ACT (fail-closed), protecting every caller against an intra-call recycle, not just the reaper.
- **Confidence: theoretical** — reaper path proven-safe; the un-pinned window is a code reading (cc-teardown's header bounds callers to desk/reaper).

**V6 · [S3] I3/I6 · seam: lead-supervisor / lead-reconciler / lead-deathwatch × launchd — watcher liveness rests on an unverified top-of-chain consumer**
- **Evidence:** each layer writes a heartbeat and declares "its ABSENCE is the alarm" (supervisor S-4 :113–120; reconciler L4-c :83–86; deathwatch L1-e :96–100). The reconciler consumes the deathwatch heartbeat (lead-reconciler:70). But **no KeepAlive launchd job** exists for supervisor/reconciler/deathwatch — only caffeinate-floor + power-policy-verify carry KeepAlive; the consumer that reads the *supervisor's* absence and pages is not co-located and unverified.
- **Mechanism:** the watcher chain terminates in a heartbeat file whose absence must be consumed by something with independent liveness (OS KeepAlive or a human). Run as `--daemon` under non-KeepAlive wiring, a crashed supervisor stays dead and its absence-alarm is never read → the fleet silently loses its past-threshold/stall/permission pager.
- **Failure scenario:** the supervisor OOMs; nothing relaunches it; hung sessions and permission prompts stop being paged; invisible until the operator notices (MEMORY: "supervisor daemon separately stale — fires zombie pages until reloaded").
- **Fix:** pin every watcher daemon under launchd **KeepAlive** (OS-level top-of-chain liveness that can't silently crash), or add an explicit heartbeat-absence consumer with its own KeepAlive — give "absence is the alarm" a named reader.
- **Confidence: theoretical** — heartbeat chain verified in code; launch/consume wiring is C10/operator and absent from the repo's `launchd/`.

## (3) MISSED-DIMENSION LIST — what a per-component review structurally cannot see

- **M1 · Metric decorrelation across the recycle/OOM boundary (I5).** A reviewer of `waiting-recycle.sh` validates a correct `used_pct` state machine; a reviewer of `cc-crash-report` validates correct size reporting. Neither can see that the **trigger axis** (`used_pct`, waiting-recycle:51) and the **failure axis** (transcript bytes, ef7f7a4) are different — only holding both at once reveals it. (→ V2)
- **M2 · Ledger-key ↔ actor-identity indirection (I8).** cc-dispatch's claim is internally consistent; cc-backlog's reap is internally consistent. The defect is that the `by` KEY one component writes (`<host>-$$`, cc-dispatch:58) names an identity the OTHER component's liveness oracle (`claimer_live`, cc-backlog:357) can't meaningfully resolve. A cross-component key-semantics mismatch, invisible within either file. (→ V1)
- **M3 · Divergent world-view SOURCES for one fleet (I7/I9).** reaper→cc-registry+it2; supervisor→`/tmp` telemetry; boundary-handoff→telemetry; cc-notify→cc-sessions+it2. Each source is valid in isolation; only a cross-component view notices they can **disagree** (registry live, telemetry absent) and that only the reaper built a self-check for its own blindness (cc-reaper:308–335 vs lead-supervisor:452–467, which has none). (→ V3)
- **M4 · Timestamp-format as a system-wide contract (I6).** Every age-gate (cc-backlog `fromdateiso8601` :420, supervisor epoch, reaper UTC) assumes all producers write the **same clock**. A single component writing local time silently corrupts every downstream age gate — each script's own date math stays self-consistent, so per-component review is blind. The scar is already in the tree (cc-reaper:67, a LOCAL-vs-UTC "false reaper DORMANT" fix).
- **M5 · Top-of-chain watcher liveness lives in launchd, not the scripts (I3).** Every daemon's "absence is the alarm" is a per-file claim; whether an OS KeepAlive or a human actually closes the loop is a cross-layer (script↔launchd↔operator) property no single-script review can assert (lead-supervisor:113 heartbeat vs the absent KeepAlive supervisor plist). (→ V6)
